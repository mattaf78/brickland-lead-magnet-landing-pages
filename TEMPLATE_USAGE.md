# Wellness Brickdown — Lead Magnet Landing Page Template

This document explains how to use this project to create additional Wellness Brickdown lead magnet landing pages. Read it before adding a new lead magnet so you understand what to create, what to leave alone, and how to test your work.

---

## 1. What This Project Is

This is a reusable landing page template for Wellness Brickdown / Brickland lead magnets. Each lead magnet is a free downloadable guide (a "field report") that captures email subscribers.

The template is built with:
- **TanStack Start** — server-side rendered React framework
- **TanStack Router** — file-based routing (routes map to files in `src/routes/`)
- **Tailwind CSS** — utility-first styling
- **Cloudflare Workers** — deployment platform
- **Lovable** — visual development environment

Every section of every page is driven by a single config object. To create a new lead magnet page, you create a new config file, a new route file, and a new folder of images. You do not touch any components.

## 3. Where LP001 Lives

LP001 — *The Ultra-Processed Food Trap* — lives at the root URL (`/`).

Its route file is `src/routes/index.tsx`. It imports its config from `src/content/leadMagnets/ultraProcessedFoodTrap.ts` and passes it to `<LeadMagnetPage config={config} />`.

---

## 4. Why LP001 Stays at /

LP001 is the first and currently only published page. Moving it away from `/` would break any existing shared links, bookmarks, or social previews that point to the root URL.

Future lead magnets (LP002, LP003 etc.) each get their own descriptive URL slug (e.g. `/sugar-spike-trap`). If a listings page is ever needed as a hub for all lead magnets, that is a separate future decision.

---

## 5. How Future LP002, LP003 etc. Should Be Added

Each new lead magnet requires three new items:

1. **A new config file** in `src/content/leadMagnets/`
2. **A new route file** in `src/routes/`
3. **A new asset folder** in `src/assets/leadMagnets/`

No existing files need to be changed. No components need to be changed. TanStack Router automatically picks up the new route file and updates the route tree on the next build or dev server start.

---

## 6. Recommended Naming Convention

Use this convention consistently so the relationship between files is always clear.

| Artefact | Pattern | LP001 example | LP002 example |
|----------|---------|---------------|---------------|
| Route file | `src/routes/{topic-slug}.tsx` | `index.tsx` *(root exception)* | `sugar-spike-trap.tsx` |
| URL path | `/{topic-slug}` | `/` *(root exception)* | `/sugar-spike-trap` |
| Config file | `src/content/leadMagnets/{camelCaseName}.ts` | `ultraProcessedFoodTrap.ts` | `sugarSpikeTrap.ts` |
| Config export | `export const {camelCaseName}: LeadMagnetConfig` | `ultraProcessedFoodTrap` | `sugarSpikeTrap` |
| Asset folder | `src/assets/leadMagnets/{topic-slug}/` | `ultra-processed/` | `sugar-spike-trap/` |
| Image files | Fixed names inside the asset folder | `cover.png`, `exposed.png`, `rebuild.png`, `whats-next.png` | same four names |

**Rules for choosing a slug:**
- Describe the topic in 2–4 words, all lowercase, words separated by hyphens
- No LP number in the URL (e.g. not `/lp002-sugar-spike`). The LP number belongs in the config's `leadMagnetId` field, where it appears in the classification bar and navbar
- Match the asset folder name to the route slug exactly

## 7. Step-by-Step Workflow for Creating a New Lead Magnet Page

### Step 1 — Prepare the images

Create a new folder: `src/assets/leadMagnets/{topic-slug}/`

Add the four page preview images inside it:
- `cover.png` — front cover of the field report
- `exposed.png` — evidence / exposed page
- `rebuild.png` — rebuild / action page
- `whats-next.png` — what happens next page

Images must exist before the config file can import them.

### Step 2 — Create the config file

Create `src/content/leadMagnets/{camelCaseName}.ts`.

Use `ultraProcessedFoodTrap.ts` as your starting template. Copy its full contents, then update every field:

- `leadMagnetId` — set to `"LP002"` (or the next number)
- `fieldReportLabel` — display label, e.g. `"Brickland Field Report"`
- `classificationBarText` — the top classification bar text
- `brandName`, `countyName` — Brickland brand values for this LP
- `hero`, `problem`, `mechanism`, `discover`, `midCta`, `preview`, `whyItMatters`, `about`, `finalCta`, `faq`, `footer`, `form` — replace all copy with the new lead magnet's content
- Image imports — update the four `import` statements at the top to point to the new asset folder
- `privacyPolicyUrl` — set to the real privacy policy URL (not `"#"`)

The TypeScript type `LeadMagnetConfig` (defined in `types.ts`) tells you exactly which fields are required and which are optional.

### Step 3 — Create the route file

Create `src/routes/{topic-slug}.tsx`.

Use `src/routes/index.tsx` as your starting template. Copy its full contents, then make these three changes:

1. Change `createFileRoute("/")` to `createFileRoute("/{topic-slug}")`
2. Change the import to bring in the new config export, e.g.:
   ```ts
   import { sugarSpikeTrap as config } from "@/content/leadMagnets/sugarSpikeTrap";
   ```
3. Update the `head()` function — change the title, description, Open Graph tags, and FAQPage structured data to match the new lead magnet

### Step 4 — Start the dev server

```
npm.cmd run dev
```

TanStack Router automatically detects the new route file and regenerates `src/routeTree.gen.ts`. You do not need to touch that file.

Open `http://localhost:8080/{topic-slug}` to see the new page.

Open `http://localhost:8080/` to confirm LP001 is unchanged.

## 8. Files That Should Usually Not Be Touched

| File / Folder | Reason |
|---------------|--------|
| `src/components/leadMagnet/` | All components are fully config-driven. Changes here affect every LP simultaneously |
| `src/components/ui/` | shadcn/ui library components. Update via the shadcn CLI only |
| `src/content/leadMagnets/types.ts` | Only change this if a new LP genuinely requires a new config field |
| `src/router.tsx` | Router setup. No changes needed |
| `src/routeTree.gen.ts` | Auto-generated. Any manual changes will be overwritten |
| `src/routes/__root.tsx` | Root layout and fallback metadata. Only change for site-wide head or layout changes |
| `vite.config.ts` | Vite/Lovable build config. Only change with care |
| `package.json`, `wrangler.jsonc` | Project and deployment config. Only change if adding a dependency or updating deployment settings |

## 9. How to Test Locally

### Run the development server

```
npm.cmd run dev
```

Opens the site at `http://localhost:8080`. Hot-reloads on file changes.

### Run a production build

```
npm.cmd run build
```

Builds to `dist/`. Use this to catch any build errors before deploying. Check that `dist/client/favicon.svg` exists and that all assets have been compiled.

## 10. How to Commit and Push

Check what has changed:
```
git status
```

Stage all changes:
```
git add .
```

Commit with a clear message:
```
git commit -m "Add LP002: Sugar Spike Trap landing page"
```

Push to the remote repository:
```
git push
```

Use descriptive commit messages that name the LP and the change, e.g.:
- `"Add LP002 config and route"`
- `"Fix LP002 hero copy"`
- `"Update og:image for LP001"`

## 11. How Lovable Sync Fits In

[Lovable](https://lovable.dev) is the visual development environment used to build and edit components in this project.

**Working locally (in Claude Code or your editor):**
- Config files, route files, and asset additions can be created directly in your local editor
- Push to the Git repository when ready
- Lovable will sync automatically from the repository

**Working in Lovable:**
- Lovable can scaffold new route files and edit component markup visually
- After making changes in Lovable, pull the latest from the repository into your local environment before making further local changes (`git pull`)

**Rule of thumb:** Do not edit the same file simultaneously in both Lovable and your local editor. Commit and sync between environments to avoid conflicts.

## 12. Pre-Publish QA Checklist

Before marking any lead magnet page as live, work through this checklist.

### Desktop
- [ ] Page renders correctly at 1280px and 1440px
- [ ] No layout overflow or horizontal scroll
- [ ] All images load
- [ ] All CTA buttons are visible and clickable
- [ ] Navbar CTA scrolls to the opt-in form

### Mobile (375px)
- [ ] Page renders correctly at 375px viewport width
- [ ] No text is cut off or overflows its container
- [ ] Images scale correctly
- [ ] Opt-in form is fully usable on a small screen
- [ ] Buttons are large enough to tap

### Images
- [ ] Cover image loads in the hero section
- [ ] All four preview images load in the Preview section
- [ ] No broken image placeholders

### Form
- [ ] Name and email fields accept input
- [ ] Consent checkbox works
- [ ] Submit button is labelled correctly
- [ ] Success state appears after submission
- [ ] Form is wired to the email service provider (see *Items Still Left for Launch*)

### FAQ
- [ ] All FAQ items expand and collapse correctly
- [ ] Answer text is accurate

### Metadata
- [ ] Browser tab shows the correct title for this LP
- [ ] Page description is accurate (check with View Source)
- [ ] `og:title`, `og:description`, and `og:image` are set correctly in the route's `head()`
- [ ] FAQPage structured data contains the correct questions and answers

### Privacy link
- [ ] `privacyPolicyUrl` in the config points to a real, working page
- [ ] Privacy link in the consent area opens correctly

### Social preview
- [ ] Test the live URL in a social preview tool (e.g. opengraph.xyz or the Facebook Sharing Debugger)
- [ ] Preview image, title, and description all display correctly
- [ ] Note: `og:image` must be an absolute URL (e.g. `https://yourdomain.com/assets/cover.png`) for social crawlers to display the image — see *Items Still Left for Launch*
